## Summary

当前变更整体方向正确，但**不建议直接合并**：存在 1 个会直接导致结果错误的关键问题（回答只取到最后一个块/最后一行）。另有若干可靠性与可维护性风险，建议一并修复。

## Critical Issues

1. **只返回最后一个答案块，导致“响应只能读到最后一行/最后一段”**  
   - 文件：`src/dom-poller.ts`  
   - 位置：`69-71`  
   - 问题：候选文本收集后使用 `response = texts[texts.length - 1]`，显式丢弃前面所有块内容。  
   - 影响：多段回答时只返回最后一段（与你反馈现象一致）。  
   - 建议：改为聚合（例如按顺序 `join("\n\n")`）并再做去噪，而不是只取最后一个元素。

## Warnings

1. **首页清理失败时静默继续，可能把旧输入与新输入拼接**  
   - 文件：`src/comet-skill.ts`  
   - 位置：`59-83`  
   - 问题：`ensureCleanHomePage()` 超时后不会抛错或降级处理，函数直接返回；后续 `fillInput()` 仅 paste 不清空，可能出现“旧 query + 新 query”拼接。  
   - 建议：超时后显式抛错，或在注入前做可验证清空策略。

2. **重连逻辑硬编码 DEFAULT_CONFIG，忽略调用方配置**  
   - 文件：`src/dom-poller.ts`  
   - 位置：`190-193`  
   - 问题：`tryReconnect()` 使用 `connect(DEFAULT_CONFIG)`，若外部传入自定义 `cdpEndpoint/timeout`，重连会偏离当前会话配置。  
   - 建议：将当前 `config` 透传到 `pollForResult()` / `tryReconnect()`。

3. **`stream-monitor` 的 `Network.dataReceived` 事件使用方式存在协议风险**  
   - 文件：`src/stream-monitor.ts`  
   - 位置：`40-58`  
   - 问题：逻辑依赖 `params.data`，但该字段在常见 CDP `Network.dataReceived` 事件中并不稳定可用（通常只有长度信息）。  
   - 影响：该模块若被重新启用，可能长期拿不到 chunk，最终只能靠 timeout 返回空结果。  
   - 建议：若保留该模块，改用明确可读 payload 的事件/接口并补集成测试；否则标注废弃或移除。

## Backward Compatibility Violations

未发现“为兼容旧行为而保留的冗余双轨逻辑”。本次改动在“直接替换旧实现”方面是干净的。

## Performance Concerns

1. **轮询脚本每次都读取整页 `innerText`，在长页面上开销较高**  
   - 文件：`src/dom-poller.ts`  
   - 位置：`13` + `16-84`（整段轮询脚本）  
   - 问题：每轮都做 `document.body.innerText` + 全局按钮/文本扫描；默认 1.5s 轮询，长时间任务下会产生不必要负担。  
   - 建议：优先缩小扫描范围（`main` 容器）、缓存稳定节点、只在状态切换时做重扫描。

## Minor Suggestions

1. `CometConfig.connectTimeoutMs` 已引入但当前未在 `connectToTarget()` 生效，可考虑接入超时控制，避免连接阶段无界等待。  
2. `src/index.ts` 文件权限变更为 `100755`（可执行）是合理的 CLI 行为，建议在团队规范里明确以避免后续 PR 误报。
