## Verdict

The fix correctly addresses the immediate symptom — `process.exit(0)` ensures the CLI always terminates. The `disconnectAll()` change is also appropriate. However, there is a **sidecar connection leak** in `dom-poller.ts` that `disconnectAll()` cannot reach, meaning the `process.exit(0)` is masking an underlying resource cleanup gap. Ship-ready as-is since `process.exit(0)` is a reliable backstop, but the sidecar leak should be addressed.

## Critical Findings

### Sidecar CDP connection is never closed

- **File:** `src/dom-poller.ts:141`
- **What:** `findAndConnectSidecar()` creates a **new** `CometClient` instance (`tempClient = createCometClient()`) that is entirely separate from the `defaultCometClient` singleton. When the agent task switches to the sidecar tab, this new client opens a WebSocket connection that is never closed — not by `disconnect()`, not by `disconnectAll()`, because those operate on `defaultCometClient`.
- **Why it matters:** Before this fix, this orphaned WebSocket was the *primary* reason agent_task hung — even a correct `await main()` with `disconnectAll()` would still leave this handle alive. After this fix, `process.exit(0)` forcibly kills it, but the connection remains leaked for the entire task duration and any cleanup logic that might be added later (graceful shutdown, connection pooling, etc.) will miss it.
- **Recommendation:** Either:
  - (a) Use `defaultCometClient.connectToTab()` instead of creating a temporary client, so the pool tracks it and `disconnectAll()` closes it; or
  - (b) Store the `tempClient` reference and explicitly close it in a cleanup path.

  Option (a) is simpler and consistent with the existing architecture.

## Strategic Concerns

### `process.exit(0)` masks future handle leaks

- **File:** `src/index.ts:119`
- **What:** `process.exit(0)` is a blunt instrument — it will hide any future event-loop leaks (timers, sockets, listeners) rather than surfacing them.
- **Why it matters:** For a CLI tool this is acceptable and pragmatic. But if comet-claw ever becomes a library imported by other tools (it already exports `executeTask`), callers won't get the `process.exit(0)` backstop and will experience the same hang.
- **Recommendation:** No action needed now. Just be aware that library consumers would need to handle their own cleanup. The `isDirectRun` guard already correctly limits `process.exit` to CLI mode.

### `stream-monitor.ts` timeout timer outlives its usefulness

- **File:** `src/stream-monitor.ts:49`
- **What:** `setTimeout(complete, config.timeout)` creates a timer that lives for the full timeout duration (60-180s). When `ac.abort()` triggers `complete()` early via the AbortSignal listener (line 114), the `clearTimeout(timeoutId)` at line 44 does clean it up. This is correct.
- **Why it matters:** No bug here. Noting for completeness — the lifecycle is sound.

## What's Good

- The fix is minimal and surgical — two lines that directly address the root cause.
- Using `.then(() => process.exit(0)).catch(() => process.exit(1))` correctly handles both success and rejection without swallowing the error (the catch block in `main()` already logs and exits with code 1 for task errors; the outer `.catch` catches truly unexpected failures).
- Changing `disconnect()` to `disconnectAll()` is the right default for a CLI exit path — there's no reason to leave connections open.

### Key Decisions Observed

1. **`process.exit(0)` over fixing all handle leaks:** Pragmatic for a CLI tool. Correct trade-off for immediate fix; sidecar leak should still be cleaned up separately.
2. **`disconnectAll()` over `disconnect()`:** Ensures all pooled connections are closed at exit rather than just the first one. Correct semantic change for the exported `disconnect()` function which is only called at CLI shutdown.
